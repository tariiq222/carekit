هيكل patient/portal/ — BFF Multi-endpoint

📱 api/mobile/patient/ — الهيكل الكامل
api/mobile/patient/ ├── bookings/ // single-BC, lightweight ├── profile/ ├── history/ ├── payments/ └── portal/ ← BFF ├── home.controller.ts // GET /mobile/patient/portal/home // → bookings+invoices+messages+profile ├── upcoming.controller.ts // GET /mobile/patient/portal/upcoming // → next 3 bookings only (fast) ├── summary.controller.ts // GET /mobile/patient/portal/summary // → outstanding invoices + last payment └── ... (screen per endpoint)
📐 قاعدة البناء — portal/ controllers
كل portal endpoint: ✓ مخصص لـ screen واحد ✓ يستدعي BCs المطلوبة فقط ✓ يُعيد response مُشكّل للـ view ✗ لا يحتوي business logic ✗ لا يكتب مباشرة للـ DB ✗ لا يُشارك response schema مع dashboard --- مثال: home vs upcoming --- home → 4 BC calls → rich payload upcoming→ 1 BC call → minimal payload الفرق يُحدد في الـ controller، مو الـ BC
