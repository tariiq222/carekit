# CareKit Design Context

## Users

**Primary users**: Receptionists and clinic administrators — not developers or executives.

- **Receptionists** use the dashboard all day, every day: booking appointments, managing patient queues, processing payments, answering phone calls while navigating the UI. Speed and clarity are survival requirements. Every extra click costs real time.
- **Clinic admins/owners** use it for oversight: reviewing reports, managing staff, configuring settings. They need confidence that the system is under control.

**Job to be done**: Get operational tasks done fast, without friction, without confusion. The UI should get out of the way and let the work happen.

**Context of use**: Busy clinic environments — multiple browser tabs open, phone ringing, patients waiting at the desk. The interface competes for attention with the physical world.

---

## Brand Personality

**Three words**: Modern. Elegant. Efficient.

The product serves Saudi clinics that want to feel premium — not just functional. CareKit should make clinic owners proud to show it to patients and partners. It should feel like it belongs in a well-designed Apple Store, not a hospital IT department.

**Emotional goals** (all four, in order of priority):

1. **Confidence and control** — "I own the situation"
2. **Speed and efficiency** — "I finish tasks in minimal steps"
3. **Elegance and professionalism** — "This system reflects well on my clinic"
4. **Ease and simplicity** — "Nothing here stresses me out"

---

## Aesthetic Direction

**Reference**: Apple Health / iOS apps — clean visual hierarchy, generous whitespace, glassmorphism used with restraint, information that surfaces exactly when needed without cognitive overload.

**White-label system**: CareKit's own brand colors are Royal Blue `#354FD8` + Lime Green `#82CC17`. These are the *default* values — every client deployment can override them via the white-label config. Always use semantic tokens (`--primary`, `--accent`, `--surface`, etc.), never hardcoded hex values. The entire frosted glass aesthetic adapts automatically to each client's palette.

**Visual system** (already implemented, must be maintained):

- Frosted glass surfaces (`backdrop-filter: blur(24px)`) — not solid white cards
- Animated gradient blobs in the background — depth, not noise
- Royal Blue `#354FD8` as CareKit's default primary — authoritative without being cold
- Apple Green `#82CC17` as CareKit's default accent — vitality, health, life
- IBM Plex Sans Arabic — RTL-first, modern, highly legible
- 8px spacing grid — rhythm and consistency throughout
- Border radius: 8 / 12 / 16 / 20px — iOS-grade softness
- Shadows: whisper-soft, never heavy

**Theme**: Light mode is default. Dark mode is a first-class citizen — ships with both. Colors and surfaces fully adapt, no hacks.

**Anti-references** (explicitly what CareKit must NOT look like):

- Legacy medical software (Clinic Master, old HIS systems): cluttered, dense, gray, dated
- ERP / SAP: complex, tabular, soul-crushing
- Generic Bootstrap templates: unstyled, impersonal, interchangeable
- Rigid Material Design: Android-flavored, wrong cultural register for Arabic healthcare

---

## Design Principles

1. **Surface the signal, hide the noise** — every screen should show exactly what the user needs right now. Secondary actions live in menus, not on the main surface.

2. **Respect Arabic-first** — RTL is not an afterthought. Layouts flow naturally in Arabic; English is an override. Spatial logic, reading direction, and typography all serve the Arabic speaker first.

3. **Glass, not plastic** — surfaces are semi-transparent and layered, creating depth. Never flat white, never opaque gray. The frosted glass aesthetic is a brand signature.

4. **Speed is a feature** — interactions must feel instant. Optimistic UI updates, skeleton loading states instead of spinners, minimal page transitions. The UI should never feel like it's making the user wait.

5. **Accessible by default, always** — target WCAG 2.1 AAA. Contrast ratios, focus states, keyboard navigation, reduced motion support — these are not optional polish, they are correctness.

6. **Tokens, not colors** — always use CSS custom properties. The white-label system depends on it; hardcoded hex values break client theming.
