# CareKit Mobile — UI Design Specification
> للمصمم: هذا الملف المرجع الكامل لتصميم تطبيق المريض وتطبيق الممارس. كل ما تحتاجه موثق هنا.

---

## 1. نظرة عامة

| البند | القيمة |
|---|---|
| المنصة | iOS + Android (React Native / Expo) |
| الاتجاه | RTL أولاً (عربي) مع دعم LTR (إنجليزي) |
| اللغات | العربية (افتراضية) + الإنجليزية |
| التطبيقات | تطبيق واحد يشمل دورين: مريض + ممارس |
| التوقيت | Asia/Riyadh |
| الـ Branding | يُجلب ديناميكياً من `GET /whitelabel/public` عند فتح التطبيق |

---

## 2. Branding & Whitelabel

الشعار والألوان والاسم **لا تُضمَّن في التصميم بشكل ثابت** — كلها تأتي من الـ API عند بدء التطبيق.

### ما يُعاد من الـ API:
```
clinic_name       ← اسم العيادة بالعربي
clinic_name_en    ← اسم العيادة بالإنجليزي
app_name          ← اسم التطبيق بالعربي
app_name_en       ← اسم التطبيق بالإنجليزي
logo_url          ← رابط الشعار
primary_color     ← اللون الأساسي (hex)
secondary_color   ← اللون الثانوي (hex)
contact_phone     ← رقم التواصل
contact_email     ← إيميل التواصل
```

### القيم الافتراضية (Fallback):
| المتغير | القيمة الافتراضية |
|---|---|
| primary_color | `#1D4ED8` (Royal Blue) |
| secondary_color | `#84CC16` (Apple Green) |
| اسم التطبيق | كيركيت / CareKit |
| الشعار | شعار CareKit الافتراضي |

> **ملاحظة للمصمم:** صمم كل الشاشات بحيث تقبل أي لون أساسي. استخدم CSS Variables أو Theme tokens وليس قيم ملونة ثابتة.

## 4. هيكل التنقل (Navigation)

### تطبيق المريض

```
App Launch
  └── GET /whitelabel/public  ← تحميل الـ Branding
  └── Check Auth Token
        ├── لا يوجد توكن → Auth Stack
        └── يوجد توكن   → Patient Tabs

Auth Stack
  ├── /login          ← تسجيل الدخول
  ├── /register       ← حساب جديد
  ├── /otp            ← تحقق OTP
  ├── /forgot         ← نسيت الباسورد
  └── /reset          ← تعيين باسورد جديد

Patient Tabs (Bottom Navigation — 5 tabs)
  ├── 🏠 الرئيسية     → home.tsx
  ├── 📅 مواعيدي     → appointments.tsx
  ├── 💬 المساعد     → chat.tsx
  ├── 🔔 الإشعارات   → notifications.tsx
  └── 👤 حسابي       → profile.tsx

Patient Modals / Stack Screens
  ├── booking/[serviceId]   ← اختيار الخدمة
  ├── booking/schedule      ← اختيار الموعد
  ├── booking/intake-form   ← نموذج ما قبل الحجز
  ├── booking/confirm       ← مراجعة وتأكيد
  ├── booking/payment       ← اختيار طريقة الدفع
  ├── booking/bank-transfer ← رفع إيصال
  ├── booking/success       ← تأكيد الحجز
  ├── appointment/[id]      ← تفاصيل الموعد
  ├── practitioner/[id]     ← ملف الممارس
  ├── rate/[bookingId]      ← تقييم الجلسة
  ├── problem-report/[id]   ← الإبلاغ عن مشكلة
  └── settings              ← الإعدادات
```

### تطبيق الممارس

```
App Launch (نفس منطق Auth)

Practitioner Tabs (Bottom Navigation — 4 tabs)
  ├── 📋 اليوم        → today.tsx
  ├── 📆 الجدول      → calendar.tsx
  ├── 👥 المرضى      → patients.tsx
  └── 👤 حسابي       → profile.tsx

Practitioner Stack Screens
  ├── appointment/[id]       ← تفاصيل + إجراءات الحجز
  ├── patient/[id]           ← ملف المريض
  ├── patient/[id]/intake    ← نماذج المريض
  ├── schedule/availability  ← إدارة التوفر
  ├── schedule/breaks        ← الاستراحات
  └── schedule/vacations     ← الإجازات
```

## 5. الشاشات التفصيلية — تطبيق المريض

### 5.1 Auth Stack

#### شاشة تسجيل الدخول `/login`
**States:** Default | Loading | Error

| العنصر | التفاصيل |
|---|---|
| الشعار | من `logo_url` — يتوسط الشاشة |
| اسم التطبيق | من `app_name` |
| حقل الإيميل | نوع `email`، RTL |
| حقل الباسورد | مع زر إظهار/إخفاء |
| زر تسجيل الدخول | Primary Button، Full Width |
| رابط "نسيت الباسورد" | نص قابل للنقر |
| رابط "تسجيل الدخول بـ OTP" | للتبديل بين الطريقتين |
| رابط "حساب جديد" | في أسفل الشاشة |

---

#### شاشة OTP `/otp`
**States:** Default | Loading | Error | Resend Available

| العنصر | التفاصيل |
|---|---|
| وصف | "أدخل الكود المرسل إلى بريدك الإلكتروني" |
| حقول OTP | 6 خانات منفصلة، LTR دائماً |
| مؤقت | عداد تنازلي 10 دقائق |
| زر "إعادة الإرسال" | يظهر بعد انتهاء المؤقت |

---

#### شاشة التسجيل `/register`

| العنصر | التفاصيل |
|---|---|
| الاسم الأول | نصي |
| الاسم الأخير | نصي |
| الإيميل | email type |
| الباسورد | مع متطلبات واضحة |
| تأكيد الباسورد | تحقق في real-time |
| الجنس | اختياري، Segmented Control (ذكر / أنثى) |
| زر التسجيل | Primary Button |

### 5.3 Booking Flow (متعدد الخطوات)

#### الخطوة 1: اختيار الخدمة + الممارس
**API:**
- `GET /services/categories`
- `GET /services`
- `GET /practitioners?serviceId=X`
- `GET /practitioners/:id/services/:serviceId/types`

| العنصر | التفاصيل |
|---|---|
| Progress Indicator | خطوة 1 من 4 |
| فلترة بالتخصص | Chips أفقية |
| بطاقة الممارس | صورة، الاسم، التخصص، التقييم، السعر |
| زر عرض الملف | → `practitioner/[id]` |

---

#### الخطوة 2: اختيار الموعد
**API:**
- `GET /practitioners/:id/slots?date=X&serviceId=Y`
- `GET /clinic/holidays`

| العنصر | التفاصيل |
|---|---|
| Progress Indicator | خطوة 2 من 4 |
| التقويم | أسبوع أو شهر، العطل مغلقة |
| شبكة المواعيد | فترات صباحية / مسائية |
| الوقت المتاح | أخضر ← متاح |
| الوقت المحجوز | رمادي ← غير متاح |
| نوع الحجز | Clinic / Phone / Video — حسب ما يدعمه الممارس |

---

#### الخطوة 3: Intake Form (إن وُجد)
**API:**
- `GET /intake-forms?scope=service&serviceId=X` أو `practitionerId=Y`
- `POST /intake-forms/:formId/responses`

| العنصر | التفاصيل |
|---|---|
| Progress Indicator | خطوة 3 من 4 |
| أنواع الحقول | نص، اختيار، متعدد، تاريخ، ملف |
| الحقول الإلزامية | علامة * واضحة |
| حفظ المسودة | auto-save أثناء الكتابة |

> **ملاحظة:** هذه الخطوة تظهر فقط إذا كان للخدمة أو الممارس نموذج `pre_booking`.

---

#### الخطوة 4: المراجعة والتأكيد
**API:** `POST /bookings`

| العنصر | التفاصيل |
|---|---|
| Progress Indicator | خطوة 4 من 4 |
| ملخص الحجز | الخدمة، الممارس، التاريخ، الوقت، الفرع |
| تطبيق كوبون | حقل كوبون + زر تطبيق (`POST /coupons/apply`) |
| بطاقة هدية | حقل كود البطاقة + رصيد (`POST /gift-cards/check-balance`) |
| ملخص السعر | السعر الأصلي، الخصم، الضريبة 15%، الإجمالي |
| زر التأكيد | → شاشة الدفع |

---

#### شاشة الدفع
**API:** `POST /payments/moyasar` أو `POST /payments/bank-transfer`

**خياران:**

**1. الدفع الإلكتروني (Moyasar):**
- حقول: رقم البطاقة، تاريخ الانتهاء، CVV
- أيقونات Visa / Mada / Mastercard
- تشفير واضح (🔒)

**2. التحويل البنكي:**
- بيانات الحساب البنكي (IBAN، اسم صاحب الحساب)
- زر رفع الإيصال ← Camera أو Photo Library
- رسالة "سيتم التحقق خلال 24 ساعة"

---

#### شاشة النجاح
| العنصر | التفاصيل |
|---|---|
| Animation | Lottie checkmark أو Success illustration |
| رقم الحجز | بارز ومنسوخ بنقرة |
| ملخص سريع | التاريخ، الوقت، الممارس |
| زر "إضافة للتقويم" | iCal / Google Calendar |
| زر "العودة للرئيسية" | |

### 5.5 آلة الحالات — Booking Status Machine

#### من منظور المريض:

| الحالة | العرض | الإجراءات المتاحة |
|---|---|---|
| `pending` | "بانتظار التأكيد" — Amber | إلغاء (مباشر إذا `patientCanCancelPending=true`) |
| `confirmed` | "مؤكد" — Green | إعادة الجدولة \| طلب إلغاء |
| `checked_in` | "وصلت للعيادة" — Teal | لا إجراء |
| `in_progress` | "الجلسة جارية" — Blue | لا إجراء |
| `completed` | "مكتمل" — Royal Blue | تقييم (إذا لم يُقيَّم) \| إعادة الحجز \| الإبلاغ عن مشكلة |
| `cancelled` | "ملغي" — Red | إعادة الحجز |
| `pending_cancellation` | "طلب إلغاء قيد المراجعة" — Orange | لا إجراء |
| `no_show` | "لم تحضر" — Gray | — |
| `expired` | "انتهت صلاحية الحجز" — Warm Gray | إعادة الحجز |

#### من منظور الممارس:

| الحالة | الإجراءات المتاحة |
|---|---|
| `pending` | تأكيد ← `POST /bookings/:id/confirm` |
| `confirmed` | Check-in ← `POST /bookings/:id/check-in` \| إلغاء |
| `checked_in` | بدء الجلسة ← `POST /bookings/:id/start` |
| `in_progress` | إنهاء الجلسة ← `POST /bookings/:id/complete` \| تسجيل غياب |
| `completed` | لا إجراء |
| `pending_cancellation` | الموافقة ← `POST /bookings/:id/cancel/approve` \| الرفض ← `POST /bookings/:id/cancel/reject` |

- استخدام SSE لـ Streaming — الرد يظهر تدريجياً حرفاً حرفاً

## 11. الـ i18n والـ RTL

### قواعد عامة:
- الاتجاه الافتراضي: **RTL (عربي)**
- الأرقام والتواريخ والمبالغ: **LTR دائماً** حتى في RTL (خط DM Sans)
- أرقام الهاتف وIBAN: LTR
- أيقونات الاتجاه (`←` `→`) تنعكس في RTL
- `padding-start` / `padding-end` بدلاً من `padding-left` / `padding-right`

### مفاتيح الترجمة الرئيسية:
```
auth.*           ← تسجيل الدخول والتسجيل
tabs.*           ← عناوين التبويبات
home.*           ← شاشة الرئيسية
appointments.*   ← المواعيد
bookings.*       ← عملية الحجز
payments.*       ← الدفع
notifications.*  ← الإشعارات
profile.*        ← الحساب الشخصي
common.*         ← نعم، لا، حفظ، إلغاء، ...
errors.*         ← رسائل الأخطاء
```

## 12. تدفق Walk-in Claim

```
  ↓

  ↓
```

## 13. التدفقات الإضافية للمريض

```

  ↓

```

```
  ↓

  ↓
عرض كل الجلسات المُنشأة
```

```
من تفاصيل موعد مكتمل ← "الإبلاغ عن مشكلة"
  ↓
اختيار نوع المشكلة (8 أنواع)
  ↓
حقل وصف نصي (اختياري)
  ↓
POST /problem-reports
```

## 14. ملاحظات تقنية للمصمم

| البند | التفصيل |
|---|---|
| Safe Area | احسب دائماً `SafeAreaInsets` (الشُقّة، شريط الـ Navigation) |
| Keyboard Avoidance | الشاشات التي تحتوي حقول + زر في الأسفل تحتاج `KeyboardAvoidingView` |
| Pull to Refresh | كل قائمة قابلة للسحب للتحديث |
| Infinite Scroll | القوائم الطويلة (مواعيد، مرضى) تستخدم pagination `page + perPage` |
| Haptics | الأزرار الرئيسية والإجراءات المهمة تحتاج Haptic Feedback |
| Loading Skeleton | استخدم Skeleton Cards وليس Spinner للقوائم |
| Error Boundary | كل قسم رئيسي له Error Boundary مستقل |
| Offline State | رسالة "لا يوجد اتصال" مع إمكانية الاستخدام المحدود |
