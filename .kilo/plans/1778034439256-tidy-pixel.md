# Zoho Invoice Integration ‚Äî Skill Plan

## 0. Answer to your rate-limit question

You asked: with shared OAuth, do all tenants' API calls count against **us**?

**No.** Zoho's API quota is **per Zoho organization** (`X-com-zoho-invoice-organizationid`), not per OAuth client. Each tenant connects their **own Zoho org**; their calls are billed against their own plan:

| Limit type | Scope | Typical value |
|---|---|---|
| Requests / minute | Per Zoho org | 100 |
| Requests / day | Per Zoho org √ó plan | Free 1k ¬∑ Standard 5k ¬∑ Premium 10k+ |
| Concurrent calls | Per Zoho org | ~10 |
| Access-token generation | Per OAuth **client** (ours) | ‚â§ 20 tokens / 10 min per refresh token; ‚â§ 15 refresh-token grants / min |

The only thing pooled on us is **token issuance**, which we eliminate with per-tenant in-memory access-token cache (Zoho access tokens last 1h). Effectively unlimited for our scale.

**Decision: Shared OAuth app + per-tenant Connect (recommended option).** One Zoho client_id/secret in env; each tenant runs the Connect flow once and we store their refresh_token + zoho_organization_id + DC.

## 1. Two billing surfaces

| Surface | Who pays whom | Zoho org owner | Deqah module |
|---|---|---|---|
| **Tenant ‚Üí Client** | Patient pays clinic | Tenant's own Zoho org | `integrations/zoho-invoice` (per-tenant config) |
| **SaaS ‚Üí Tenant** | Tenant pays Deqah (subscription invoices) | **Deqah's** Zoho org | `platform/saas-billing/zoho` (single global config in env) |

Subscriptions/dunning logic stays in Deqah (`platform/billing`, already exists ‚Äî reused). Zoho is **strictly downstream**: it only reads from Deqah, issues a PDF/hosted invoice, and returns a link. Zoho never mutates Deqah business state. No Zoho recurring profiles ‚Äî we POST a fresh invoice each cycle.

**Currency**: SAR only for v1. We reject Connect if the chosen Zoho organization's base currency ‚âÝ SAR (clear error: "Zoho org currency must be SAR").

## 3. Data model

### 3.1 `Integration` row (no schema change needed)

`provider = "zoho-invoice"`, `config` JSON (encrypted ciphertext) holds:
```ts
{
  refreshToken: string,
  zohoOrganizationId: string,   // Zoho's org_id, sent in X-com-zoho-invoice-organizationid
  dataCenter: 'com'|'sa'|'eu'|'in'|'au'|'jp'|'ca',
  webhookSecret: string,        // generated by us, given to tenant to paste in Zoho webhook config
  defaults: {
    itemId?: string,            // default Zoho item for booking line-items
    branchId?: string,
    sendOnCreate: boolean,
    paymentTerms?: string,
  }
}
```

### 3.2 New table: `ZohoInvoiceLink` (Prisma schema/integrations.prisma ‚Äî new file)

Maps Deqah invoices ‚Üî Zoho invoices for idempotency + status sync.
```prisma
model ZohoInvoiceLink {
  id                 String   @id @default(uuid())
  organizationId     String   // Deqah tenant
  scope              String   // 'TENANT_CLIENT' | 'SAAS_TENANT'
  deqahInvoiceId     String?  // FK to local Invoice
  deqahBookingId     String?
  zohoInvoiceId      String   @unique
  zohoCustomerId     String
  zohoOrganizationId String
  status             String   // mirrors Zoho status: draft|sent|paid|partially_paid|void|overdue
  total              Decimal
  currency           String   // 'SAR' for v1
  invoiceUrl         String?  // Zoho hosted payment page (we share with the client)
  pdfUrl             String?
  createdAt          DateTime @default(now())
  updatedAt          DateTime @updatedAt

  @@unique([organizationId, deqahInvoiceId])
  @@index([organizationId, status])
  @@index([zohoCustomerId])
}

model ZohoContactLink {
  id              String   @id @default(uuid())
  organizationId  String
  deqahPersonId   String
  zohoContactId   String
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt

  @@unique([organizationId, deqahPersonId])
  @@unique([organizationId, zohoContactId])
}

model ZohoWebhookEvent {   // dedup + audit
  id             String   @id @default(uuid())
  organizationId String
  eventId        String   // Zoho event id
  eventType      String
  payload        Json
  processedAt    DateTime?
  createdAt      DateTime @default(now())

  @@unique([organizationId, eventId])
}
```

For SaaS‚Üítenant scope, `deqahInvoiceId` references the platform-billing invoice and `organizationId` = platform org id, with `scope='SAAS_TENANT'` as the discriminator.

```ts
  jp:  { accounts: 'https://accounts.zoho.jp',  api: 'https://www.zohoapis.jp' },
  ca:  { accounts: 'https://accounts.zohocloud.ca', api: 'https://www.zohoapis.ca' },

```

### 4.2 Connect flow
1. `GET /dashboard/integrations/zoho/connect?dc=sa` ‚Üí
   builds `https://accounts.zoho.{dc}/oauth/v2/auth?scope=ZohoInvoice.fullaccess.all&client_id=...&response_type=code&access_type=offline&prompt=consent&redirect_uri=https://api.deqah.app/dashboard/integrations/zoho/callback&state=<signed{orgId,dc,nonce}>`.
2. User consents in Zoho ‚Üí redirected back with `code`, `location` (DC), `accounts-server`.
3. Callback exchanges `code` for `{access_token, refresh_token}` against `accounts.zoho.{dc}/oauth/v2/token`.
4. Calls `GET zohoapis.{dc}/invoice/v3/organizations` to list available Zoho orgs; if >1 we redirect to a "Pick Zoho organization" screen, else auto-select.
5. Generates a webhook secret, stores everything encrypted in `Integration.config`, returns webhook URL + secret to display in dashboard.

### 4.3 Token cache
Same shape as `ZoomApiClient.tokenCache` keyed by `orgId:fingerprint(refreshToken)`. Refresh proactively 60s before expiry. Invalidate on disconnect.

### 4.4 Required scopes
`ZohoInvoice.fullaccess.all` (or narrower: `ZohoInvoice.invoices.ALL ZohoInvoice.contacts.ALL ZohoInvoice.customerpayments.ALL ZohoInvoice.creditnotes.ALL ZohoInvoice.settings.READ`).

## 5. Sync flows

### 5.1 Tenant ‚Üí Client invoicing

**Trigger**: **after payment**, on `payment.captured` from the Moyasar webhook. We never invoice up-front.

```
moyasar.payment.captured
  ‚Üí ensure ZohoContactLink for the paying client (create via POST /contacts if missing)
  ‚Üí POST /invoices  (status will start as "sent" since it's already paid)
       line_items derived from booking.serviceVariants, currency = SAR
  ‚Üí POST /invoices/{id}/payments    // record the Moyasar capture as customerpayment
       payment_mode='creditcard', reference=<moyasar_charge_id>, amount=full
  ‚Üí store ZohoInvoiceLink with status='paid' and invoice_url + pdf_url
  ‚Üí emit deqah event `zoho.invoice.issued` carrying {invoiceUrl, pdfUrl} so the
    notification module can deliver the link to the client (email/WhatsApp).
```

The dashboard also exposes the Zoho invoice URL on the booking detail page. The tenant can resend manually from there.

**Refunds**: when the existing refund flow settles a Moyasar refund ‚Üí `POST /creditnotes` referencing the original `zohoInvoiceId` ‚Üí `POST /creditnotes/{id}/refunds` with the Moyasar refund id as reference. Stored on `ZohoInvoiceLink` for audit.

### 5.2 SaaS ‚Üí Tenant billing

**Trigger**: existing `platform/billing` cycle (confirmed exists ‚Äî we reuse its "subscription period due" event/handler hook). It computes the amount due per tenant per period.

Subscription state machine **stays in Deqah**. Zoho only issues the invoice document; payment of the SaaS subscription happens through Deqah's own Moyasar checkout (same as tenant‚Üíclient). After our Moyasar capture confirms, we mirror the captured payment into the platform Zoho org:

```
platform-billing.period-due (tenant T, amount A)
  ‚Üí upsert tenant T as Zoho contact in Deqah's platform Zoho org
  ‚Üí POST /invoices (uses platform Zoho creds from env), currency=SAR
  ‚Üí store ZohoInvoiceLink with scope=SAAS_TENANT, deqahInvoiceId = platform invoice id
  ‚Üí return invoice_url + pdf_url to the platform-billing module so it can
    surface them in the admin dashboard and tenant-facing billing screen
  ‚Üí tenant pays via Deqah's Moyasar checkout ‚Üí on payment.captured we POST
    /invoices/{id}/payments to mark the Zoho copy paid (mirror only).
  ‚Üí platform-billing marks the subscription period paid using its **own**
    Moyasar signal ‚Äî never the Zoho webhook.
```

Env vars: `ZOHO_PLATFORM_CLIENT_ID`, `ZOHO_PLATFORM_CLIENT_SECRET`, `ZOHO_PLATFORM_REFRESH_TOKEN`, `ZOHO_PLATFORM_ORG_ID`, `ZOHO_PLATFORM_DC`, `ZOHO_PLATFORM_WEBHOOK_SECRET`.

### 5.3 Webhook ingestion (mirror-only, never mutates Deqah state)

Because Zoho is strictly downstream, webhooks are **metadata-only**: they update fields on `ZohoInvoiceLink` (status, viewed_at, last_email_sent_at) so the dashboard can show "Tenant viewed the invoice in Zoho", but they **never** write to Booking / Payment / Refund / Subscription tables. Deqah's source of truth is Moyasar + its own state machine.

`POST /public/webhooks/zoho/:tenantToken`

`tenantToken` = HMAC of `organizationId` so the URL is opaque but we can resolve the tenant without DB lookup before signature verification.

Steps:
1. Resolve tenant from token ‚Üí load `Integration.config`.
2. Verify signature: Zoho signs payload using the `webhookSecret` we generated; compare HMAC-SHA256 of raw body with `X-Zoho-Webhook-Signature` header (constant-time).
3. Dedup via `ZohoWebhookEvent.eventId`.
4. Dispatch on `event_type` ‚Äî **only** updates the Zoho mirror row:
   - `invoice.sent` / `invoice.viewed` ‚Üí update `ZohoInvoiceLink` viewed/sent timestamps.
   - `invoice.thankyou` / `invoice.paid` ‚Üí update `ZohoInvoiceLink.status='paid'` (audit only ‚Äî Deqah already knew this from Moyasar).
   - `invoice.void` ‚Üí update `ZohoInvoiceLink.status='void'`.
   - All other event types ‚Üí store in `ZohoWebhookEvent` and ignore.
5. Return 200 quickly. Processing is light enough to be inline.

For the SaaS scope, the webhook URL is fixed (`/public/webhooks/zoho/platform`) and uses the platform secret. Same mirror-only rule.

**This webhook surface is optional** ‚Äî the integration works without it. Tenants who don't configure the webhook in Zoho still get fully working invoicing; they just lose the "invoice viewed" UX in the dashboard.

## 7. Resilience

- **Retry/backoff**: copy `fetchWithRetry` pattern from Zoom client, treat 429 specially by reading `Retry-After` header.
- **Idempotency**: `ZohoInvoiceLink` unique on `(organizationId, deqahInvoiceId)` prevents duplicate Zoho invoices if our handler retries.
- **Outbox**: emitting Zoho calls from event handlers should write to an outbox row first (or use the existing event-bus' at-least-once semantics) so a Zoho outage doesn't lose invoices. Reuse the same pattern Zoom booking handler uses; if Zoom doesn't have outbox, we add a minimal `IntegrationOutbox` table just for Zoho.
- **Token rotation**: refresh tokens are long-lived but can be revoked; on `INVALID_TOKEN` (Zoho code 57) mark integration `isActive=false` and surface a banner in the dashboard prompting reconnect.
- **DC mismatch**: if callback `location` header differs from selected `dc`, trust the header and persist that.

## 8. Security

- All config encrypted at rest (reuse `ZoomCredentialsService` pattern; create `ZohoCredentialsService` with the same envelope encryption keyed per-org).
- Webhook secret is per-tenant, 32-byte random, displayed once in the dashboard.
- OAuth `state` is signed (HS256) with `{orgId, dc, nonce, exp}`, 5-min TTL.
- Rate-limit our webhook endpoint at the gateway layer (1 RPS/tenant burst 20).
- Audit: log every mutating Zoho call to `IntegrationAuditLog` (existing? else new) with redacted bodies.

## 11. Locked decisions

1. **Product name**: Deqah (not CareKit). All tables/types use `deqah*` prefix where Deqah-side ids are referenced.
2. **SaaS billing**: existing `platform/billing` module is reused. We hook into its "subscription period due" event; we do **not** introduce a new scheduler.
3. **Invoice trigger**: **after payment only** ‚Äî on Moyasar `payment.captured`. We never create Zoho invoices before payment. The Zoho invoice is created and immediately marked paid in the same operation.
4. **Currency**: SAR only for v1. Connect step rejects Zoho orgs whose base currency ‚âÝ SAR with a clear error message.
5. **Direction**: Zoho is strictly **read-only / downstream**. It reads from Deqah, issues the invoice document/PDF, and returns a hosted-invoice URL. Webhooks from Zoho update only the `ZohoInvoiceLink` mirror row (status/viewed timestamps); they never touch Booking, Payment, Refund, or Subscription tables. Deqah + Moyasar remain the single source of truth for money state.
