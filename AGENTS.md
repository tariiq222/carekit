# CareKit — Agent Instructions

Built with NestJS + Prisma (backend), React Native/Expo (mobile), Next.js 14 (dashboard), and OpenRouter AI (chatbot).

## Maestro Responsibilities

The `maestro` is the team orchestrator. Responsibilities:

1. **Plan** — Break down milestones into tasks for each agent
2. **Delegate** — Assign tasks via inbox messages with full context
3. **Unblock** — Resolve cross-agent conflicts and architectural decisions
4. **Review** — Final check on all deliverables before marking done
5. **Sync** — Keep all agents aligned on shared types, API contracts, and schema changes

The user (Tariq) reviews maestro's results and gives direction. Maestro does NOT need permission for routine task delegation.

---

## Mandatory Laws (Always Enforced)

These apply to ALL agents without exception:

1. **No code before schema** — Prisma schema must be finalized and approved by maestro before any backend code is written
2. **No secrets in code** — All API keys, passwords, connection strings go in `.env` only. Never in source code.
3. **TypeScript strict** — Zero `any` types allowed. Strict mode on everywhere.
4. **Swagger first** — Every API endpoint must be documented in Swagger before it's considered done
5. **RTL-first UI** — Every UI component must support RTL layout. Arabic is primary, not an afterthought.
6. **i18n mandatory** — No hardcoded strings. All text through i18n (nestjs-i18n / next-intl / i18next)
7. **No direct DB access** — All database operations go through Prisma. No raw SQL except for migrations.
8. **Conventional Commits** — All commit messages follow: `feat:`, `fix:`, `chore:`, `docs:`, `test:`, `refactor:`
9. **PR required** — No direct push to `main`. All changes via PR, reviewed by maestro.
10. **Definition of Done** — A task is NOT done until: TypeScript passes, tests exist, Swagger documented, RTL tested, Arabic+English work.

## Schema Change Protocol

`backend/prisma/schema.prisma` is the most critical shared file. ANY change follows this exact protocol:

1. Agent proposes change → sends to maestro with justification
2. Maestro reviews impact on: backend, mobile, dashboard
3. Maestro notifies ALL affected agents BEFORE approving
4. backend-dev executes migration (`prisma migrate dev`)
5. ALL agents update their types from `shared/` after migration
6. No agent writes code against the OLD schema after migration

---

## Shared Types Workflow

All TypeScript types are centralized in `shared/` to prevent duplication:

- All types for API requests/responses live in `shared/types/`
- Database types auto-generated by Prisma (`prisma generate`) → exported to `shared/`
- API DTOs defined in `backend/`, then exported to `shared/`
- `mobile/` + `dashboard/` import from `shared/` — NEVER define duplicate types
- When schema changes → `prisma generate` → update `shared/` → notify all agents

## Database Migration Protocol (Mandatory)

### Rules — ALL agents MUST follow

1. **NEVER use `prisma db push`** — it changes the database without a migration record. Always use `prisma migrate dev --name <descriptive_name>`.

2. **Migration naming convention:**

   ```bash
   prisma migrate dev --name <action>_<what>
   ```

   Examples:
   - `--name init` (first migration)
   - `--name add_activity_log`
   - `--name add_notes_to_booking`
   - `--name rename_price_to_amount`
   - `--name add_index_on_booking_date`

3. **One migration per change** — Don't batch unrelated changes. One PR = one migration max.

4. **Migrations are IMMUTABLE** — Once a migration is committed to Git, NEVER edit it. Create a new migration instead.

5. **Migration files go to Git** — `prisma/migrations/` folder is tracked. Every commit that changes schema.prisma MUST include the migration file.

6. **Schema Change Flow:**

   ```text
   Agent proposes schema change → message to maestro
   Maestro approves (checks impact on all agents)
   backend-dev edits schema.prisma
   backend-dev runs: npx prisma migrate dev --name <name>
   backend-dev runs: npx prisma generate (updates types)
   backend-dev commits: schema.prisma + migration file + generated types
   Maestro notifies ALL agents: "Schema updated — pull latest"
   All agents pull + run: npx prisma generate (to get updated types)
   ```

7. **ONLY backend-dev touches schema.prisma** — No other agent edits the schema directly. All changes go through the Schema Change Flow above.

8. **Production deployment:**

   ```bash
   # On client server — applies all pending migrations safely
   npx prisma migrate deploy
   ```

   NEVER run `prisma migrate dev` on production.

9. **Rollback strategy:**
   - If a migration breaks something → create a NEW migration that reverts the change
   - NEVER delete a migration file from Git history
   - Example: `prisma migrate dev --name revert_add_notes_to_booking`

10. **Migration log (maintained by maestro):**
    Keep a running log in `docs/migration-log.md`:

    ```markdown
    # CareKit Migration Log

    | # | Date | Migration Name | Description | Agent | PR |
    |---|------|---------------|-------------|-------|-----|
    | 1 | 2026-03-22 | init | Initial schema — 25 models | backend-dev | #1 |
    | 2 | 2026-03-25 | add_activity_log | ActivityLog model for audit trail | backend-dev | #12 |
    ```

## Forbidden Patterns

- `AsyncStorage` for sensitive data → use `expo-secure-store`
- `any` type in TypeScript → define proper types
- Hardcoded API URLs or secrets → use env vars
- Direct database queries outside Prisma → use Prisma client
- Skipping Swagger documentation → document before closing task
- Non-conventional commit messages → enforce `feat:`, `fix:`, etc.
- UI components without RTL support → RTL is required, not optional
- Forms without Zod validation → all forms need schema validation
- Duplicate types in mobile/dashboard → import from `shared/` only
- Direct push to `main` → always via PR

## Key Project Decisions (Non-Negotiable)

- **Authentication:** Email+password OR Email+OTP — NO SMS
- **Authorization:** CASL permission-based RBAC — 5 default roles + custom roles from dashboard
- **Payments:** Moyasar (primary) + Bank transfer with AI vision verification
- **Video:** Zoom API auto-generated meeting links only
- **Storage:** MinIO self-hosted (not AWS S3)
- **Deployment:** Docker per client on Dokploy
- **Money:** Stored as integers (halalat), converted on frontend only
- **Dates:** All UTC, timezone conversion on frontend only
- **Cancellation:** Manual admin decision per case — no automatic refund rules
